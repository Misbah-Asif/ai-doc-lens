# Product Requirements Document

## Product Name

**AI Doc Lens**

## Document Version

Version 1.0

## Product Type

Open-source Python document intelligence and compliance validation engine.

## Status

Planned MVP

---

# 1. Product Overview

AI Doc Lens is an open-source tool that extracts structured information from business documents using an LLM and validates the extracted information against deterministic compliance rules.

The first version will support salary slips in text-based PDF format.

The system will:

1. Accept a document.
2. Extract readable text from the document.
3. Extract predefined fields using an LLM.
4. Validate the extracted response using Pydantic.
5. Run deterministic Python compliance rules.
6. Return a structured compliance result.

The LLM will only extract information.

The final compliance decision will be made by Python rules.

```text
AI extracts facts.
Python rules evaluate compliance.
Humans review uncertainty.
```

---

# 2. Problem Statement

Businesses regularly receive documents such as:

* Salary slips
* Bank statements
* Identity documents
* Invoices
* Tax documents
* Employment documents
* Vendor certificates

These documents are often checked manually for:

* Required fields
* Valid dates
* Monetary values
* Missing information
* Internal policy requirements
* Document freshness
* Data consistency

Manual document review is repetitive, slow, and inconsistent.

Traditional automation struggles because document formats vary between companies and issuers.

AI Doc Lens solves this by combining:

* LLM-based document understanding
* Structured schema validation
* Deterministic compliance rules


The salary-slip extractor will return:

```text
employee_name
```

Optional future fields:

```text
employee_id
designation
basic_salary
deductions
```

## 6.4 Compliance Checks

The MVP will include:

1. Employee name must be present.
2. Employer name must be present.
3. Salary month must be present.
4. Salary slip must not be older than three months.
5. Gross salary must be greater than zero.
6. Net salary must be greater than zero.
7. Net salary must not be greater than gross salary.
8. Currency should be identifiable.
9. Ambiguous or incomplete extraction may require manual review.

# 7. Functional Requirements

## FR-1: PDF Input

The system must accept a local PDF file path through the Python SDK and CLI.

The API must accept a PDF through multipart form-data.

## FR-2: File Validation

The system must validate:

* File exists
* File is readable
* File extension is PDF
* PDF can be opened
* PDF contains extractable text

## FR-3: Text Extraction

The system must extract text from all PDF pages.

The system must preserve reasonable page separation.

The system must return a clear error if no readable text exists.

## FR-4: Document Type Selection

The user must provide a document type in MVP.

Example:

```text
salary_slip
```

Automatic classification is outside MVP scope.

## FR-5: LLM-Based Field Extraction

The system must send the extracted document text and document-specific instructions to an LLM.

The LLM must return data matching the expected document schema.

The prompt must instruct the model:

* Do not guess
* Return null for missing data
* Return valid JSON only
* Use standardized date formats
* Use numeric monetary values

## FR-6: Schema Validation

The LLM response must be validated through Pydantic.

Invalid responses must produce a structured extraction error.

## FR-7: Compliance Rule Execution

The system must run the registered compliance rules for the selected document type.

Each rule must return:

```text
rule identifier
status
value
reason
evidence
```

## FR-8: Final Status

The final status must be calculated using the following priority:

```text
If any check is FAIL:
    Final status = FAIL

Else if any check is MANUAL_REVIEW:
    Final status = MANUAL_REVIEW

Else:
    Final status = PASS
```

## FR-9: Structured Output

The output must be JSON serializable.

## FR-10: Python SDK

The package must expose a simple public interface.

Example:

```python
from aidoclens import validate_document

result = validate_document(
    path="salary-slip.pdf",
    document_type="salary_slip",
)
```

## FR-11: CLI

The CLI must support:

```bash
aidoclens validate salary-slip.pdf --type salary_slip
```

Optional command:

```bash
aidoclens list-document-types
```

## FR-12: API

The FastAPI application must expose:

```text
GET /health
GET /v1/document-types
POST /v1/documents/validate
```

## FR-13: LLM Provider Abstraction

The core engine must not depend directly on a single LLM vendor.

The project must define an LLM provider interface.

The MVP may implement one provider first.

## FR-14: Fake Provider for Tests

The project must include a fake LLM provider for unit and integration testing.

Tests must not depend on a live external LLM API.

---

# 8. Non-Functional Requirements

## NFR-1: Privacy

Uploaded files must not be permanently stored.

Temporary files must be deleted after processing.

The project must not log full document contents.

## NFR-2: Extensibility

Adding a new document type should not require modifying the core extraction pipeline.

## NFR-3: Testability

Compliance rules must be independently testable.

The LLM layer must be replaceable with a fake provider.

## NFR-4: Reliability

Malformed LLM responses must not crash the application.

Unsupported files must return clear errors.

## NFR-5: Maintainability

Code must be modular and typed.

Functions should have narrow responsibilities.

## NFR-6: Security

The system must:

* Reject unsupported file types
* Avoid exposing stack traces through APIs
* Limit uploaded file size
* Avoid storing secrets in code
* Use environment variables for API credentials

## NFR-7: Performance

For small text-based PDFs, processing should complete within a reasonable interactive request duration.

Most latency will depend on the selected LLM provider.

## NFR-8: Open-Source Readiness

The repository must include:

* README
* LICENSE
* CONTRIBUTING guide
* Example environment file
* Tests
* Sample synthetic document
* Clear extension instructions

# 13. Module Responsibilities

## 13.1 `models/compliance.py`

Contains shared output models:

* CheckStatus
* ComplianceCheck
* ComplianceResult
* ErrorResponse

## 13.2 `readers/`

Responsible only for reading files and extracting raw text.

The reader must not:

* Call the LLM
* Apply rules
* Determine compliance

## 13.3 `llm/`

Contains the provider abstraction and implementations.

Responsibilities:

* Send prompts
* Receive model responses
* Parse structured output
* Raise provider-specific errors as common exceptions

## 13.4 `documents/`

Contains document-specific logic.

Each document type contains:

```text
schema.py
prompt.py
rules.py
config.py
```

## 13.5 `engine/`

Contains reusable workflow logic.

Responsibilities:

* Registry lookup
* Prompt formatting
* Extraction
* Schema validation
* Rule execution
* Final result construction

## 13.6 `api/`

Contains only HTTP-specific concerns.

Responsibilities:

* File upload
* Request validation
* Temporary file handling
* Mapping exceptions to HTTP responses

## 13.7 `cli.py`

Contains terminal command definitions.

Business logic must remain inside the engine, not the CLI.

# 14. Core Data Models

## 14.1 Status Enum

```python
from enum import Enum


class CheckStatus(str, Enum):
    PASS = "PASS"
    FAIL = "FAIL"
    MANUAL_REVIEW = "MANUAL_REVIEW"
```

## 14.2 Compliance Check

```python
from typing import Any

from pydantic import BaseModel


class ComplianceCheck(BaseModel):
    rule: str
    status: CheckStatus
    value: Any | None = None
    reason: str | None = None
    evidence: str | None = None
```

## 14.3 Compliance Result

```python
class ComplianceResult(BaseModel):
    document_type: str
    status: CheckStatus
    extracted_data: dict
    checks: list[ComplianceCheck]
```

## 14.4 Salary Slip Schema

```python
from datetime import date
from decimal import Decimal

from pydantic import BaseModel, Field


class SalarySlipData(BaseModel):
    employee_name: str | None = None
    employer_name: str | None = None
    salary_month: date | None = None
    gross_salary: Decimal | None = Field(default=None, ge=0)
    net_salary: Decimal | None = Field(default=None, ge=0)
    currency: str | None = None
```

# 16. LLM Provider Interface

```python
from abc import ABC, abstractmethod

from pydantic import BaseModel


class LLMProvider(ABC):

    @abstractmethod
    def extract(
        self,
        prompt: str,
        response_model: type[BaseModel],
```

# 17. Prompt Requirements

The salary-slip prompt must instruct the model:

```text
You are extracting structured data from a salary slip.

Do not guess missing values.
Return null when information is not present.
Return only valid JSON.
Use YYYY-MM-01 for salary_month.
Return monetary values as numbers only.
Do not add fields outside the schema.
```

Example expected schema:

```json
{
  "employee_name": null,
  "employer_name": null,
  "salary_month": null,
  "gross_salary": null,
  "net_salary": null,
  "currency": null
}
```

The document text will be appended after the instructions.

# 18. Compliance Rule Design

Each rule must be:

* Independent
* Deterministic
* Small
* Unit testable
* Free from direct LLM calls
* Free from file-reading logic

Example:

```python
def employee_name_present(
    data: SalarySlipData,
) -> ComplianceCheck:
    if not data.employee_name:
        return ComplianceCheck(
            rule="employee_name_present",
            status=CheckStatus.FAIL,
            reason="Employee name is missing.",
        )

    return ComplianceCheck(
        rule="employee_name_present",
        status=CheckStatus.PASS,
        value=data.employee_name,
    )
```

## Required Rule Functions

```text
employee_name_present
employer_name_present
salary_month_present
salary_slip_is_recent
gross_salary_is_positive
net_salary_is_positive
net_salary_not_greater_than_gross
currency_is_present
```

# 19. Manual Review Rules

MANUAL_REVIEW should be used when the system cannot confidently determine whether a condition passes or fails.

Possible cases:

* Multiple salary months detected
* Multiple employee names detected
* Gross salary cannot be distinguished from total earnings
* Currency is missing but inferred from symbols
* Document text is unusually short
* LLM returns incomplete but partially usable data
* Date format is ambiguous

The MVP may implement limited manual-review logic.

Deterministic missing required fields may still return FAIL.

# 27. Fake LLM Provider

The project must include a deterministic fake provider.

Example:

```python
class FakeLLMProvider(LLMProvider):

    def __init__(self, response: dict):
        self.response = response

    def extract(
        self,
        prompt: str,
        response_model: type[BaseModel],
    ) -> BaseModel:
        return response_model.model_validate(self.response)
```

This allows testing without:

* API cost
* Network access
* Non-deterministic model responses
* Provider rate limits

# 40. Key Product Principle

The most important architectural rule is:

```text
Do not use the LLM as the compliance authority.
```

The LLM should:

```text
Read
Understand
Extract
Normalize
```

The Python engine should:

```text
Validate
Compare
Apply policies
Determine status
Explain failures
```

This separation makes AI Doc Lens easier to test, trust, extend, and use in real systems.
The clean MVP flow is therefore:

  Validate PDF
      ↓
  Extract page text using pypdf
      ↓
  Reject if no meaningful text exists
      ↓
  Send extracted text and schema instructions to LLM
      ↓
  Validate LLM JSON with Pydantic
      ↓
  Apply deterministic Python rules
